The single best heuristic I've found for picking a stack is: **can I debug this at 3am, alone, with no internet?**

Not "is it fast." Not "is it scalable." Not "does it have a nice ergonomic API." Those things matter, but they don't matter at 3am, when the only thing that matters is whether you can read the stack trace.

Here is what I currently use, and why.

## Postgres

Postgres is my therapist.

There is no situation that another database does better for my workloads. Time-series? `tsrange` and partitioning. Full-text search? Built in. Queue? `SELECT FOR UPDATE SKIP LOCKED`. JSON blobs? Yes, even if it's not pretty. Geospatial? PostGIS.

The reason isn't that Postgres is the best at any one of those. It's that the failure modes are *legible*. When Postgres breaks at 3am, the error message says what's wrong. When MongoDB broke at 3am in 2018, the error message said "?".

I have, over six years, had to learn maybe forty Postgres failure modes. I now know roughly thirty-five of them on sight. The remaining five are why I have a notebook.

## TypeScript

I write TypeScript for the same reason. Not because the types are great (they're fine), but because *I can read the stack trace*. When a thing breaks in production, I can usually guess within ten lines where the bug is, and within ten more I can fix it.

A language I write 40 hours a week is a language whose failure modes I've internalized. That's the only thing that matters.

## SQLite (the weird one)

The exception to the boring-stack rule is SQLite, which is not boring — it's *aggressively* clever — but it's so legible that it earns the spot anyway.

I use SQLite for:

- Local prototypes (because it's a file)
- Small services that need a database but not a DBA (because it's a file)
- The CI dashboard side project (because it's a file)
- The diff-club newsletter backend (because it's a file)

I do not use SQLite for anything that needs concurrent writes from more than ~10 actors, or anything where I need a separate read replica, or anything where the storage layer is going to live across more than one machine.

But the moment SQLite fits, it really fits. The whole database is a file you can `scp`. You can read the schema with `.schema`. You can fix a bad row with `UPDATE`. There's nothing to fail except your own SQL.

## What I avoid

I avoid, in roughly this order:

- **NoSQL databases that aren't a cache.** If I want JSON in Postgres, I'll use the JSON column. If I want a cache, I'll use Redis. Anything in between, I don't trust.
- **Microservices for teams smaller than 30.** The complexity is real, the benefits are theoretical.
- **GraphQL.** I don't have a strong argument here. It's just a vibe. Three projects, three times I wished I'd shipped REST.
- **Anything that requires a Kubernetes cluster on day one.** Day one is when you have one user (you) and you don't need orchestration.

## The 3am test

Imagine you get paged at 3am. The site is down. You have a laptop, a phone with bad reception, and 20 minutes before things get worse.

Now look at each piece of your stack and ask: *can I debug this right now?*

If the answer is yes, keep it. If the answer is "I'd have to look something up in the docs," consider what you're trading away. Maybe the trade is worth it — sometimes the right tool is the unfamiliar one. But know what you're trading.

I've lost more 3ams to tools that were "fast" than to tools that were boring. The boring ones I just *fixed*. The fast ones I had to argue with.

## So pick boring

Pick boring, write the README, document the failure modes, and save your weirdness for the parts of the product that the user actually sees.

You can be weird in the UI. Be boring underneath.
